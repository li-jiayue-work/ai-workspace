# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 工作区概览

这是一个多项目工作区（workspace root），CLI 工具在任意子目录执行时会自动载入对应目录的 CLAUDE.md，无需手动指定。

| 目录 | 用途 |
|------|------|
| `active/` | 活跃项目，当前在开发 |
| `archive/` | 已完成或暂停的项目，保留代码参考 |
| `experiments/` | 实验、原型、探索性代码 |
| `templates/` | 项目模板、配置模板 |
| `agents/` | 自定义 Agent、Workflow 定义、MCP 配置 |
| `snippets/` | 跨项目可复用的代码片段 |
| `learning/` | 学习笔记 |
| `assets/` | 图片、素材等静态资源 |
| `memory/` | 跨会话持久记忆（用户偏好、项目上下文、反馈） |

## 用户偏好

- 主要语言：Python，但也用 Go、TypeScript 等，根据项目选最合适的
- 项目类型不限：全栈 Web、后端 API、AI/ML、脚本工具等
- 偏好务实方案，不做过早抽象，不过度设计
- 代码缩进见 `.editorconfig`：代码文件 4 空格，配置/JSON/CSS 2 空格，Makefile 用 tab

## 工作区约定

- 新项目建在 `active/` 下，复制 `templates/project/CLAUDE.md` 作为起点并填充项目具体信息
- 项目暂停或完成后，将目录移到 `archive/` 下，保留代码参考
- 实验性代码放在 `experiments/` 下，不需要模板
- 子目录的 CLAUDE.md 会自动加载，覆盖根目录的对应配置
- `memory/` 目录存储跨会话持久记忆，由 Claude Code 自动管理，无需手动编辑

## 行为准则（基于 Andrej Karpathy 对 LLM 编码问题的观察）

以下四条原则旨在减少常见的 LLM 编码失误。对于简单任务可灵活判断，不必过度教条。

### 1. 先思考，再动手

**不要假设，不要隐藏困惑，主动展示权衡。**

动手编码前：
- 明确陈述你的假设。如果不确定，提出来问。
- 如果存在多种理解方式，列出来——不要悄悄选择。
- 如果有更简单的方案，直接说出来。该 push back 时就 push back。
- 如果某件事不清楚，停下来，指出困惑点，提问。

### 2. 简洁第一

**用最少的代码解决问题。不做揣测性开发。**

- 不添加超出需求的功能。
- 不为单次使用的代码创建抽象。
- 不添加未被要求的「灵活性」或「可配置性」。
- 不为不可能发生的场景做错误处理。
- 如果写了 200 行但其实 50 行就够了，重写它。

自问：「一个资深工程师看到这个会说过度复杂吗？」如果是，简化。

### 3. 外科手术式的修改

**只碰必须改的。只清理自己造成的烂摊子。**

编辑已有代码时：
- 不要「顺便优化」相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 即使你的风格偏好不同，也要匹配现有风格。
- 如果你注意到无关的死代码，提出来——但不要删。

当你的修改产生孤儿代码时：
- 删除**你的修改**导致的未使用 import / 变量 / 函数。
- 不要删除之前就存在的死代码，除非明确要求。

检验标准：每一个修改行都应该能直接追溯到用户的请求。

### 4. 目标驱动的执行

**定义成功标准。循环执行直到验证通过。**

将任务转化为可验证的目标：
- 「加校验」→「为无效输入写测试，然后让测试通过」
- 「修 bug」→「写一个能复现 bug 的测试，然后让它通过」
- 「重构 X」→「确保重构前后测试都通过」

对于多步骤任务，给出简短的计划：

```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

强成功标准让你能独立循环推进。弱标准（「把它搞对」）需要反复沟通确认。

---

**判断这些准则是否有效：** diff 中不必要的改动减少、因过度复杂导致重写的次数减少、澄清性问题出现在实施之前而非在错误之后。
